5

Behaviour, routing & states

What each door does, and the states 1486 must handle.
Two doors, one fork
Create project (kept) → the shipped modal, manual name/identifier, unchanged. Plan a new project with AI (new) → /onboarding (the fork). The two are independent affordances; neither is nested in the other.
The fresh-vs-import fork is on /onboarding only (MOTIR-1461). 1486 does not build a second fork; nothing here connects a repo or picks a tracker.
Create-project states (composed, unchanged)
Loading — the “Create project” button spins + disables both buttons while the Server Action runs (shipped isPending). Error — an identifier collision surfaces inline (IDENTIFIER_COLLISION); any other failure is a toast. Both exist today (Panel 3).
Empty (no projects) → Panel 2a: both doors, primary “Plan a new project with AI” + secondary “Create project”. The switcher trigger reads “No project”.
“Already has AI plan”
If the active project already has a planning session (an AiPreplanSession / generated tree), the empty state (Panel 2b) swaps “Plan a new project with AI” → “Continue your plan”, routing to the resume surface rather than restarting onboarding. We never re-onboard a project that is already planned.
Placement & keyboard
Switcher: “Plan a new project with AI” is the accent row directly above “Create project”, both below the hairline. Empty state: the AI button is the accent primary, “Create project” the secondary. Each is a focusable button in normal tab order; Enter / Space activates.
