The reusable AI planning workspace — shell + universal entrance

MOTIR-1193 · 7.20.1 — the ONE shared planning interface. Every AI-planning surface (generation review 7.4, re-planning 7.11, contextual planning 7.12, the persistent roadmap 7.19, and onboarding 7.3) composes the same full-screen canvas (left) + chat (right) shell — they are MODES (states) of this one surface, not separate UIs. This asset designs the workspace shell + its behaviours + the universal entrance. The canvas pane is the standalone WorkItemCanvas (MOTIR-1194; design @ design/roadmap/) — composed here, never redrawn.

On-canvas incremental placement. Proposed work items appear on the standalone canvas one by one as the conductor proposes them, each drawn with its relationship edges — parent→child, the within-story depends_on arrow, and the cross-story blocked_by warning edge — using the canvas’s own node + edge language (the StationCard + PlanningCanvas conventions from design/roadmap/). Confirm-to-persist: the proposed set is a STATE of the canvas — pending nodes (dashed accent border + a proposed pill) and pending edges (dashed). Nothing is written to the DB until the user presses Confirm; Discard drops the whole proposal. This is the diff/review surface — no separate review screen.

One surface, four states. Generation (7.4), augment/re-plan (7.11), contextual (7.12) and roadmap-read (7.19) differ only in (a) what the canvas is seeded with — empty vs an existing tree vs one item’s subtree — and (b) the chat driver’s framing. The shell, the on-canvas placement, and the confirm-to-persist gate are identical. Onboarding (7.3) is the one specialization that wraps this shell in its gated per-tier pre-plan review loop (see onboarding.mock.html). Each state is reached through the entrance door called out above — drawn in sheet 4.

One affordance — same everywhere

Consistency. Same label (“Plan with AI”), same ✦ icon, same accent treatment, same placement language on every surface — a primary button in toolbars / nav, a menu item in the ⋯ menu and ⌘K. Built once as the reusable PlanWithAILauncher (MOTIR-1299, blocked_by this design); each surface mounts it, the detail (910) + roadmap (1011) doors adopt it. Composes components/ui/* (Button / Popover / CommandPalette) — --el-* colour + semantic shape tokens only; AA + keyboard-reachable.

The door determines the MODE. From a specific work item → contextual (7.12); from a project surface with no item → re-plan (7.11), or generation (7.4) if no plan exists; from the roadmap → roadmap read + augment (7.19). The launcher passes the originating context to the workspace, which opens in the matching state from sheet 3.
