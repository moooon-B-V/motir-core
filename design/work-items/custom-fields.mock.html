Custom fields on the issue detail rail

Story 5.3 · design for 5.3.7 — custom-field cards, per-type inline editors, and the "Show more fields" disclosure. Composed from the shipped FieldCard / Input / DatePicker / Combobox grammars; no new chrome.

0 · Placement — custom cards after Estimate, before created/updated

Custom-field cards are a contiguous block in definition position order, immediately after the Estimate card (the last built-in FieldCard) and before the created/updated audit list. Fields with values render as cards; fields without values collapse behind the "Show more fields (N)" disclosure that sits after the valued cards (the verified Jira hide-when-empty rule). Built-ins never reorder. Here: Severity / Customer / Effort hold values; Go-live and Stakeholder are empty, so N = 2.

1 · Per-type value rendering (display mode)

Every custom card is a FieldCard verbatim — uppercase field label + corner chevron + value line. text truncates (full value on title); number renders the formatted decimal; date reuses the Due-date grammar (muted calendar glyph + formatDate); select renders the option label plain — an archived option keeps rendering with the muted italic (archived) mark; user reuses the assignee grammar (24px initial Avatar + name). The empty placeholder is the rail's muted-italic None (the Parent card's convention).

2 · Per-type inline editors (chevron open, rotated 180°)

The chevron toggles the value line into the field's editor — the FieldCard pattern. text / number use Input chrome and commit on blur or chevron collapse (the Estimate grammar); date opens the shipped DatePicker (autoOpen; the calendar popover's authority is datepicker.mock.html) with the trigger × Clear; select opens a Combobox (autoOpen; searchable at ≥8 options) whose first row is None — the clear path — and whose menu excludes archived options; user is the AssigneePicker-shaped member Combobox (searchable, None first, email secondaries). An invalid commit (422) keeps the editor open: error border + the rose-tint message box (role="alert"). A pending write disables the editor (opacity 50%).

3 · "Show more fields" — collapsed · expanded · no-fields null case

Fields without values collapse behind a quiet full-width text button at the custom block's end — "Show more fields (N)", chevron down. Expanding (chevron rotates, label flips to "Show fewer fields", aria-expanded) reveals the empty cards, each editable in place; a field that gains a value moves above the line on refresh, and a cleared one moves back below. The row exists for every role — it is how all fields are viewed. With no custom fields defined, nothing renders at all: no disclosure, no section gap — the rail is byte-identical to today.

4 · Viewer read-only · loading skeleton

A read-only actor (project viewer, or a member on a limited project) sees values with no chevron — FieldCard editable={false}, the rail's existing read-only grammar; the disclosure row stays (viewing, not editing). The loading skeleton extends naturally: ghost label + ghost value bars in card chrome, one per custom field, same as the built-ins.

For 5.3.7: the custom block reads the bounded customFields array from getIssueDetail (definitions in position order, each with its resolved value) and commits through the dedicated setCustomFieldValueAction with router.refresh() on success — the rail's existing pattern. The disclosure is a real <button aria-expanded>; every editor is labelled by its field label (aria-label="Edit/Close <label>" on the chevron — the FieldCard grammar); errors are role="alert" rose-tint boxes. Colour only via --el-*; shape only via the element shape tokens. The per-type tile glyphs + tints (shared map with 5.3.4: type/sky · hash/peach · calendar/mint · square-chevron-down/lavender · circle-user/rose) belong to the Fields ADMIN page; the rail keeps value lines clean (calendar glyph for date, Avatar for user — the built-ins' own grammar).
