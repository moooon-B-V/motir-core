4 · Virtualization + keyboard / aria-sort model — virtualization has NO distinct visual (only viewport rows mount); this panel documents it for 2.5.14

Virtualization — invisible by design

A deep forest can hold thousands of expanded rows. Like Jira's Structure / Advanced Roadmaps grids, the treegrid only mounts the rows in (or near) the viewport — rows scrolled out of view are removed from the DOM and a spacer preserves the scroll height. There is no distinct visual: a virtualized row looks identical to a non-virtualized one. The diagram below is schematic only — the middle band is what's actually in the DOM.

A11y holds across virtualization. Each mounted row keeps its true aria-level, aria-posinset, and aria-setsize (e.g. posinset 19 / setsize 128) so a screen reader announces the real position even though only a window of rows exists — the lazy-read returns the per-node total for aria-setsize. The roving-tabindex keyboard model from the shipped TreeTable is unchanged: ↑/↓ move the active row (auto-scrolling, which mounts the landed row), → expands a collapsed parent (triggering its lazy fetch) or steps into its children, ← collapses or steps to the parent, Enter activates the row link.

Sortable headers — the affordance & a11y

Each tree-grid column header is a sort button carrying a caret hidden until hover (faint) or active (solid). The active column's role="columnheader" carries aria-sort="ascending|descending"; every other header is aria-sort="none". Direction is conveyed by the caret glyph and the aria-sort text — never colour alone (WCAG 1.4.1 / finding #35). Clicking a header sorts by that column asc; clicking the active header toggles asc↔desc; clicking another moves the active sort there. Single-column sort only (multi-sort is Epic 6). Crucially, sorting re-orders siblings WITHIN each parent — each lazy children-fetch carries the same ORDER BY, so the hierarchy is preserved; no row ever leaves its parent. Default sort is key asc (the Title column).
