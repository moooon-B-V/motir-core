Attachments panel + preview lightbox

Story 5.2 · design Subtask 5.2.4 → blocks 5.2.5 (the panel) and 5.2.6 (the lightbox). Fills the detail page's reserved Epic-5 slot — detail.pen draws no attachments surface. Jira-mirror decisions drawn here: strip + list views with a count, embeds-are-attachments with the editor-sourced delete BLOCK, the three permissions, image/PDF preview only, hard delete; deviation: cursor-paged 50 + "Show more (N)" instead of Jira's load-all-then-degrade (finding #57). Built from the real --el-* + element-shape tokens; no new primitive.

1 · Strip card anatomy — image · glyph · editor-sourced (delete BLOCKED) · no-permission
Cards are 148px-min tiles: 88px thumbnail (image cover-fit, or the MIME-family glyph on --el-surface-soft) over name + date. Download / delete reveal on hover/focus-within (drawn forced-visible here). Editor-sourced cards carry the lavender source chip and their delete is disabled with the points-at-source tooltip — the Jira rule that prevents the broken-embed hole. A card whose delete the caller lacks (not the uploader, not an admin) omits the control entirely — absent, never disabled (the 5.1 role grammar). Activating a previewable card opens the lightbox (panel 6); a non-previewable card downloads.

2 · List view — the same rows densified (adds size + uploader)
The toggle flips the SAME data into rows: family glyph · name · source chip (when editor-sourced) · size · date · uploader · the hover actions. Row click previews (image / PDF) or downloads — same split as the cards. The chosen view persists per user (the 5.1.5 sort-preference pattern). Sizes are right-aligned tabular; the glyph map is the design-notes table.

3 · Upload — drag-over dropzone · uploading queue · the three inline typed errors

5 · Delete confirm — names the file, states the hard-delete truth
The RemoveLinkButton confirm-Popover pattern verbatim (300px, --radius-card + --shadow-elevated, ghost Cancel + danger Delete), anchored to the card/row delete control. Delete is permanent — row + blob + a History "attachment removed" entry; no trash, so the copy says so. Focus returns to the panel after.

6 · Preview lightbox — image (contain-fit) · PDF (embedded frame + fallback)
A full-screen Modal (the 2.5.19 size-token growth pattern adds the full size; the overlay deepens to bg-black/80 for the lightbox) over ONE focused attachment. Header: filename + size · Download · close ×. Images render contain-fit on the scrim; PDFs in an embedded <object> frame with the can't-inline fallback message. Non-previewable types (zip, docs, csv) NEVER open it — the card downloads instead (the DTO's isImage/isPdf split). NO prev/next navigation (unverified in the mirror — documented out). Esc closes; focus returns to the opening card; the dialog is labelled by the filename.

7 · Viewer (read-only) — no Attach, no delete, no dropzone
A project viewer (6.4 read-only role) sees the panel and can preview + download, but every mutating affordance is omitted — no Attach button, no dropzone reaction, no delete controls (absent, never disabled — the same grammar as the 5.1 comments viewer state). The count and view toggle remain.
