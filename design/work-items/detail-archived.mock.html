Read-only posture (decision): the page stays fully editable — the banner is the only signal; inline controls are NOT dimmed/disabled. Motir's archive is a reversible, single-node, editor-level action (the Linear shape), so a global read-lock would add a disabled state to every field editor for something the user undoes in one click. Justified deviation from Jira (which makes archived issues read-only — but Jira's archive is a heavier admin lifecycle). See design-notes for the rationale + the documented future option.

3 · View-only (canBrowse, NOT canEdit) — the banner stays (an archived item is still readable), but the Restore button is HIDDEN, not shown-disabled
This work item is archived Archived by Bo Philips · Jun 14, 2026 · It's hidden from boards and lists but kept with its history.

Restore is gated on canEdit (the unarchiveWorkItem DELETE /api/work-items/[id]/archive is assertCanEdit-gated). A viewer who can browse but not edit sees the banner WITHOUT the Restore button — hidden, never shown-disabled (the WorkItemActionsMenu hidden-not-disabled pattern, mirroring the 2.9 list view's dropped Restore column). The "restore it to bring it back" tail of the copy is also dropped, since this viewer can't.
