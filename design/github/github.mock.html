GitHub integration — connect / settings · repo selection · PR & CI status

Story 7.10 · MOTIR-889 (design gate). The GitHub surfaces for the status-sync + review loop: a workspace connects to GitHub through the verified GitHub-App model — two independent grants (know my identity · pick my repos) — after which linked pull requests surface their PR state and CI result directly on the work item. Placement is Settings → Workspace → GitHub (the installation is workspace-scoped). Composed only from shipped primitives — Card, Pill, Button, EmptyState, Switch, the settings-area shell — with colour via --el-* tokens and shape via element-semantic tokens. Five panels: (1) not connected · (2) connected + repos · (3) PR/CI on the issue · (4) empty + revoked-error states · (5) the explicit item→PR link (MOTIR-1595) — the manual override of the auto-resolver, on the detail page's Development card.

Connect this workspace to GitHub to sync pull-request and CI status onto your work items, feed the code graph, and drive the review loop.

Connect GitHub

Two quick grants — both happen on GitHub.

Step 1 · Identity

Verify your GitHub identity

Authorize Motir to confirm who you are on GitHub — your username and avatar. This reads your public profile only. It grants no access to any code.

Step 2 · Repository access

Install the Motir GitHub App

Choose which repositories Motir may read — you pick the exact repos on GitHub’s install screen. Motir never sees the rest, and you can change the selection any time.

You’ll be sent to GitHub to authorize, then to pick repositories.
