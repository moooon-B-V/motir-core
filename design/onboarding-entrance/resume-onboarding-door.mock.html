Resume onboarding — the labeled app-shell re-entry door

A dedicated, clearly-labeled “Resume onboarding” entry point in the signed-in app shell, shown ONLY when the active project has an in-progress (un-finished) onboarding session. Activating it re-enters /onboarding, which already resumes to the real persisted step (MOTIR-1487’s “Resuming…” machinery). It is distinct from — and co-exists with — the generic “Plan with AI” launcher, and honours MOTIR-1461’s decision that the entrance front door itself carries no resume affordance.

The Resume onboarding row is a shipped SidebarItem pushed to the front of the hasProject primary section in SidebarNav.tsx, in an accent treatment (a --el-tint-lavender fill, --el-accent-on-surface icon, a trailing accent status dot). Glyph is lucide History — the same glyph the wz-bar’s “Save & exit” uses (MOTIR-1488), so the two ends of the save→resume loop read as one gesture. Href /onboarding. It is distinct from the generic PlanWithAILauncher pill (top nav / FAB / ⌘K, Sparkles): the pill is the always-on generative entrance (start / re-plan); this row is the conditional, stateful “continue what you started” door. The row does not take the pill’s hero gradient — that stays reserved for the launcher. In-progress is shown by the trailing accent dot (a text “In progress” chip truncates the 17-char label at the 240px rail width); the authoritative non-visual signal is the row’s conditional presence + aria-label="Resume onboarding (in progress)". The explicit chip appears only where there is room — the ⌘K palette below.

3 · The hidden state — a project with NO in-progress onboarding (never started, or already finished). The row is absent; only the generic “Plan with AI” launcher remains.

The row is a conditional island: it renders only when the active project’s onboarding is in progress — project.onboardingRanAt == null (server, cheap) AND GET /api/ai/pre-plan returns session != null with status !== 'tiers_complete'. A never-started project (session: null) and a finished one (onboardingRanAt set) both hide it. When hidden, nothing is added — the always-on “Plan with AI” launcher is the only planning door.

The four never appear together: Save & exit is in-session only; Resume onboarding and Continue your plan are split by onboardingRanAt (un-finished vs finished); Plan with AI is the always-on generic launcher they all sit beside. MOTIR-1461 deliberately places NO resume affordance on the /onboarding entrance itself — resume lives in the app shell (here) and in-session (1488), never on the front door.
